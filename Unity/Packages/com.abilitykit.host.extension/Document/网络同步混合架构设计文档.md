# Ability-Kit 网络同步混合架构设计文档

> **阅读对象**：希望了解 Ability-Kit 网络同步整体架构的开发者
>
> **文档目标**：让你理解"帧同步和状态同步的适用场景"、"如何根据游戏类型选择同步方案"、"混合架构如何实现"

---

## 一、设计理念：为什么需要混合架构？

### 1.1 单一方案的局限性

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     单一同步方案的局限性                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【纯帧同步的问题】                                                      │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │                                                                  │  │
│  │  1. 带宽随玩家数量线性增长                                        │  │
│  │     每个玩家的输入都需要广播给所有其他玩家                        │  │
│  │                                                                  │  │
│  │  2. 断线重连成本高                                                │  │
│  │     需要下载完整的输入历史，然后重演所有帧                        │  │
│  │                                                                  │  │
│  │  3. 作弊检测困难                                                  │  │
│  │     服务器不计算逻辑，难以检测作弊                                │  │
│  │                                                                  │  │
│  │  4. 非确定性系统难以同步                                          │  │
│  │     物理引擎、浮点运算可能导致不一致                              │  │
│  │                                                                  │  │
│  └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
│  【纯状态同步的问题】                                                    │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │                                                                  │  │
│  │  1. 延迟感知明显                                                  │  │
│  │     玩家操作后需要等待服务器状态返回                              │  │
│  │                                                                  │  │
│  │  2. 带宽消耗大                                                    │  │
│  │     每帧需要传输所有实体状态                                      │  │
│  │                                                                  │  │
│  │  3. 录像体积大                                                    │  │
│  │     录像需要存储每一帧的状态                                      │  │
│  │                                                                  │  │
│  │  4. 服务器压力大                                                  │  │
│  │     服务器需要计算所有游戏逻辑                                    │  │
│  │                                                                  │  │
│  └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 1.3 典型游戏类型的同步方案

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    典型游戏类型的同步方案选择                            │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │ 【MOBA / 格斗游戏】                                              │  │
│  │                                                                  │  │
│  │  推荐方案：帧同步 + 服务器校验                                    │  │
│  │                                                                  │  │
│  │  帧同步：                                                         │  │
│  │  - 英雄移动（确定性位移）                                         │  │
│  │  - 技能释放（确定性技能逻辑）                                     │  │
│  │  - 动画播放（同步动画状态）                                       │  │
│  │                                                                  │  │
│  │  状态同步（校验）：                                                │  │
│  │  - 伤害计算（服务器验证伤害是否合理）                             │  │
│  │  - Buff 效果（防止无限Buff）                                     │  │
│  │  - 经济变化（防止作弊刷钱）                                       │  │
│  │                                                                  │  │
│  └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │ 【FPS / TPS 射击游戏】                                           │  │
│  │                                                                  │  │
│  │  推荐方案：状态同步 + 延迟补偿                                     │  │
│  │                                                                  │  │
│  │  状态同步：                                                         │  │
│  │  - 玩家位置和旋转（服务器权威）                                   │  │
│  │  - 射击命中判定（服务器验证）                                     │  │
│  │  - 生命值和弹药（服务器校验）                                     │  │
│  │                                                                  │  │
│  │  延迟补偿：                                                         │  │
│  │  - 客户端本地预测移动                                             │  │
│  │  - 命中检测延迟补偿                                               │  │
│  │  - 击中反馈延迟显示                                               │  │
│  │                                                                  │  │
│  └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │ 【RTS / 策略游戏】                                               │  │
│  │                                                                  │  │
│  │  推荐方案：纯帧同步                                               │  │
│  │                                                                  │  │
│  │  帧同步：                                                         │  │
│  │  - 单位移动（确定性寻路）                                         │  │
│  │  - 战斗结算（确定性伤害公式）                                     │  │
│  │  - 资源采集（确定性采集逻辑）                                     │  │
│  │                                                                  │  │
│  │  优势：                                                           │  │
│  │  - 录像体积小（只存输入）                                         │  │
│  │  - 回放质量高（完全重演）                                         │  │
│  │  - 服务器压力小（只转发输入）                                     │  │
│  │                                                                  │  │
│  └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │ 【MMORPG / 大型多人】                                             │  │
│  │                                                                  │  │
│  │  推荐方案：区域化状态同步                                         │  │
│  │                                                                  │  │
│  │  状态同步：                                                         │  │
│  │  - 玩家状态（服务器权威）                                         │  │
│  │  - NPC AI（服务器计算）                                           │  │
│  │  - 物品交易（服务器校验）                                         │  │
│  │                                                                  │  │
│  │  区域化：                                                           │  │
│  │  - 只同步视野范围内的实体                                         │  │
│  │  - 减少带宽消耗                                                   │  │
│  │  - 支持大规模玩家同屏                                             │  │
│  │                                                                  │  │
│  └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 六、传输层适配

### 6.1 战斗传输层

```csharp
namespace AbilityKit.Ability.Game.Battle.Transport
{
    /// <summary>
    /// 战斗逻辑传输接口
    /// </summary>
    public interface IBattleLogicTransport
    {
        /// <summary>
        /// 帧推送事件
        /// </summary>
        event Action<FramePacket> FramePushed;

        /// <summary>
        /// 连接
        /// </summary>
        void Connect();

        /// <summary>
        /// 断开连接
        /// </summary>
        void Disconnect();

        /// <summary>
        /// 发送创建世界请求
        /// </summary>
        void SendCreateWorld(CreateWorldRequest request);

        /// <summary>
        /// 发送加入世界请求
        /// </summary>
        void SendJoin(JoinWorldRequest request);

        /// <summary>
        /// 发送离开世界请求
        /// </summary>
        void SendLeave(LeaveWorldRequest request);

        /// <summary>
        /// 发送输入
        /// </summary>
        void SendInput(SubmitInputRequest request);
    }

    /// <summary>
    /// 网关战斗逻辑传输实现
    /// </summary>
    public sealed class GatewayBattleLogicTransport : IBattleLogicTransport
    {
        private readonly GatewayBattleLogicTransportOptions _options;
        private readonly IConnection _connection;

        public event Action<FramePacket> FramePushed;

        public GatewayBattleLogicTransport(
            GatewayBattleLogicTransportOptions options,
            IConnection connection)
        {
            _options = options;
            _connection = connection;

            _connection.ServerPushReceived += OnServerPush;
        }

        private void OnServerPush(uint opCode, ArraySegment<byte> payload)
        {
            if (opCode == _options.OpFramePushed)
            {
                var packet = _options.DeserializeFramePushed.Invoke(payload);
                FramePushed?.Invoke(packet);
            }
        }

        public void SendInput(SubmitInputRequest request)
        {
            var payload = _options.SerializeSubmitInput.Invoke(request);
            _connection.Send(_options.OpSubmitInput, payload);
        }
    }

    /// <summary>
    /// 传输选项
    /// </summary>
    public sealed class GatewayBattleLogicTransportOptions
    {
        public string Host { get; set; }
        public int Port { get; set; }
        public Func<ITransport> TransportFactory { get; set; }

        // 操作码
        public uint OpCreateWorld { get; set; }
        public uint OpJoinWorld { get; set; }
        public uint OpLeaveWorld { get; set; }
        public uint OpSubmitInput { get; set; }
        public uint OpFramePushed { get; set; }

        // 序列化器
        public Func<ArraySegment<byte>, FramePacket> DeserializeFramePushed { get; set; }
        public Func<SubmitInputRequest, byte[]> SerializeSubmitInput { get; set; }
    }
}
```

## 七、设计总结

### 7.1 核心优势

| 特性 | 说明 |
|------|------|
| **灵活性** | 根据游戏类型选择帧同步、状态同步或混合方案 |
| **模块化** | 帧同步和状态同步组件独立，可按需组合 |
| **可扩展** | 通过接口和属性系统轻松添加新的同步类型 |
| **一致性** | 统一的架构支持不同的游戏类型 |
| **可测试** | 同步逻辑与游戏逻辑分离，易于单元测试 |

### 7.2 扩展点

1. **新的同步模式**：扩展 `WorldSyncMode` 枚举和对应实现
2. **新的快照类型**：实现 `IWorldStateSnapshotProvider` 并注册
3. **新的传输协议**：实现 `IBattleLogicTransport` 接口
4. **新的校验策略**：扩展 `IRollbackStateProvider` 或快照校验逻辑

### 7.3 游戏类型推荐配置

| 游戏类型 | SyncMode | InputMode | InputDelay | SnapshotInterval |
|----------|----------|-----------|------------|------------------|
| MOBA | Hybrid | Predicted | 2-3帧 | 1帧 |
| 格斗游戏 | FrameLockstep | Predicted | 1帧 | 5帧 |
| FPS/TPS | StateSynchronization | Immediate | 5-10帧 | 1帧 |
| RTS | FrameLockstep | Delayed | 0帧 | 10帧 |
| MMORPG | AuthoritativeServer | Immediate | 0帧 | 1帧 |
