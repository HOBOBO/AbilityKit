## 一、设计理念：为什么要用分层状态机管理游戏流程？

### 1.1 传统做法的游戏流程问题

```
❌ 传统游戏流程的问题：

1. 流程逻辑堆在一个类里
   ┌─────────────────────────────────────────────────────────┐
   │ void Update() {                                        │
   │     switch (_currentState) {                           │
   │         case State.Boot: DoBoot(); break;             │
   │         case State.Lobby: DoLobby(); break;           │
   │         case State.Battle: DoBattle(); break;         │
   │         case State.Result: DoResult(); break;         │
   │         // 每个状态要写一堆代码                       │
   │     }                                                 │
   │ }                                                     │
   │                                                          │
   │ // 问题：所有状态逻辑都在一个类里                     │
   │ // 改一个状态要编译整个项目                          │
   │ // 难以测试                                          │
   └─────────────────────────────────────────────────────────┘

2. 状态切换难以管理
   ┌─────────────────────────────────────────────────────────┐
   │ // 状态切换时：                                        │
   │ // - 要清理旧状态资源                                 │
   │ // - 要初始化新状态资源                               │
   │ // - 要处理中断恢复                                   │
   │ // 这些逻辑混在一起，很难维护                        │
   └─────────────────────────────────────────────────────────┘

3. 功能难以复用
   ┌─────────────────────────────────────────────────────────┐
   │ // 战斗中的调试功能，别的状态也想用？                 │
   │ // 复制粘贴？                                        │
   │ // 很难在状态间共享功能                              │
   └─────────────────────────────────────────────────────────┘

4. 状态内部嵌套复杂
   ┌─────────────────────────────────────────────────────────┐
   │ // Battle 状态下：                                    │
   │ // - Prepare (准备)                                  │
   │ // - Connect (连接)                                  │
   │ // - LoadAssets (加载资源)                           │
   │ // - InMatch (战斗中)                               │
   │ // - End (结束)                                     │
   │                                                          │
   │ // 这些子状态也是一坨代码                            │
   └─────────────────────────────────────────────────────────┘
```

### 1.2 分层状态机的解决方案

```
✅ 分层状态机的设计思路：

┌─────────────────────────────────────────────────────────────┐
│                       核心思想                              │
│                                                             │
│   把"游戏流程"变成"状态的树"                             │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐  │
│   │  传统方式：                                          │  │
│   │  ┌───────────────────────────────────────────┐    │  │
│   │   │  Game.Update()                           │    │  │
│   │   │  {                                       │    │  │
│   │   │      switch(state) {                    │    │  │
│   │   │          case Boot: DoBoot();           │    │  │
│   │   │          case Lobby: DoLobby();         │    │  │
│   │   │          case Battle: DoBattle();       │    │  │
│   │   │      }                                   │    │  │
│   │   │  }                                       │    │  │
│   │   └───────────────────────────────────────────┘    │  │
│   │   // 所有逻辑在一个方法里                         │  │
│   │                                                       │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐  │
│   │  分层状态机方式：                                    │  │
│   │  ┌───────────────────────────────────────────┐    │  │
│   │   │  RootStateMachine                        │    │  │
│   │   │  ├── Boot ─────────────────────────────  │    │  │
│   │   │  ├── Lobby ────────────────────────────  │    │  │
│   │   │  │                                         │    │  │
│   │   │  └── Battle                               │    │  │
│   │   │      ├── Prepare                          │    │  │
│   │   │      ├── Connect                           │    │  │
│   │   │      ├── LoadAssets                       │    │  │
│   │   │      ├── InMatch ──────────────────────   │    │  │
│   │   │      │    ├── Input                       │    │  │
│   │   │      │    ├── Sync                        │    │  │
│   │   │      │    └── View                       │    │  │
│   │   │      └── End                              │    │  │
│   │   │                                         │    │  │
│   │   └───────────────────────────────────────────┘    │  │
│   │   // 状态变成树，每个节点职责单一                  │  │
│   │                                                       │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
│   好处：                                                    │
│   - 状态逻辑分散，易于维护                                │
│   - 子状态可以复用父状态的功能                            │
│   - Feature 机制支持功能模块化                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 核心理念

```
┌─────────────────────────────────────────────────────────────┐
│                    游戏流程的三大承诺                          │
│                                                             │
│   1️⃣ 分层                                              │
│      游戏流程分成多层状态，每层专注自己的职责               │
│                                                             │
│   2️⃣ 可组合                                            │
│      Feature 机制让功能模块可以自由组合                    │
│                                                             │

│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、核心概念：从零理解游戏流程系统

### 2.1 什么是"游戏流程（GameFlow）"？

**通俗解释**：GameFlow 是"游戏的故事线"，管理游戏从启动到结束的整个过程。

```
┌─────────────────────────────────────────────────────────────┐
│                    GameFlow = 游戏的故事线                   │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐  │
│   │  游戏的生命周期                                      │  │
│   │                                                       │  │
│   │  Boot ──► Lobby ──► Battle ──► ... ──► End        │  │
│   │   │        │        │                        │        │  │
│   │   │        │        │                        ▼        │  │
│   │   │        │        │                     回到 Lobby  │  │
│   │   │        │        │                                   │  │
│   │   │        │        ▼                                   │  │
│   │   │        │    战斗中的子状态：                       │  │
│   │   │        │    Prepare ──► InMatch ──► End           │  │
│   │   │        │                                     │    │  │
│   │   │        ▼                                     │    │  │
│   │   │    大厅界面                                  │    │  │
│   │   │                                             │    │  │
│   │   ▼                                             │    │  │
│   │  启动界面                                        │    │  │
│   │                                                       │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 关键名词解释

| 名词 | 通俗解释 | 类比 |
|------|----------|------|
| **GameFlowDomain** | 根状态机 | 故事大纲 |
| **RootState** | 顶层状态 | 故事章节 |
| **BattleState** | 战斗子状态 | 章节中的场景 |
| **IGamePhaseFeature** | 可挂载功能模块 | 场景中的道具 |
| **BattleContext** | 战斗上下文 | 场景的道具清单 |
| **BattleSessionFeature** | 战斗会话核心 | 场景的核心道具 |

## 六、使用指南

### 6.1 添加新状态示例

```csharp
// 场景：在 Battle 中添加新的"Loading"状态

// 1. 在 BattleState 枚举中添加
private enum BattleState
{
    Prepare = 0,
    Loading = 1,  // 新增
    Connect = 2,
    // ...
}

// 2. 添加状态定义
_fsm.AddState(BattleState.Loading,
    onEnter: ctx =>
    {
        // 挂载必要的 Feature
        Attach(new LoadingScreenFeature());
    },
    onExit: ctx =>
    {
        // 卸载 Feature
        Detach<LoadingScreenFeature>();
    }
);

// 3. 添加转换
_fsm.AddTransition(BattleState.Prepare, BattleState.Loading,
    condition: ctx => ctx.ReadyToLoad);

_fsm.AddTransition(BattleState.Loading, BattleState.Connect,
    condition: ctx => ctx.LoadingComplete);
```

### 6.2 添加新 Feature 示例

```csharp
// 场景：添加一个"暂停菜单"功能

public sealed class PauseMenuFeature : IGamePhaseFeature, IOnGUIFeature
{
    private bool _isOpen;
    private Texture2D _backgroundTexture;

    public void OnAttach(in GamePhaseContext ctx)
    {
        // 加载资源
        _backgroundTexture = Resources.Load<Texture2D>("pause_bg");

        // 订阅事件
        ctx.Root.TryGetFeature(out BattleContext context);
        context.Events.Subscribe(InputEvents.EscapePressed, OnEscapePressed);
    }

    public void Tick(in GamePhaseContext ctx, float dt)
    {
        if (_isOpen)
        {
            // 更新逻辑
        }
    }

    public void OnGUI(in GamePhaseContext ctx)
    {
        if (!_isOpen) return;

        // 绘制暂停菜单
        GUI.DrawTexture(new Rect(0, 0, Screen.width, Screen.height), _backgroundTexture);
        if (GUI.Button(new Rect(100, 100, 200, 50), "Resume"))
        {
            _isOpen = false;
        }
    }

    public void OnDetach(in GamePhaseContext ctx)
    {
        // 释放资源
        Resources.UnloadAsset(_backgroundTexture);

        // 取消订阅
        ctx.Root.TryGetFeature(out BattleContext context);
        context.Events.Unsubscribe(InputEvents.EscapePressed, OnEscapePressed);
    }

    private void OnEscapePressed(object sender, EventArgs e)
    {
        _isOpen = !_isOpen;
    }
}

// 使用：挂载到需要暂停菜单的状态
_fsm.AddState(BattleState.InMatch, onEnter: _ =>
{
    Attach(new PauseMenuFeature());  // 添加这行
    // ... 其他 Feature
});
```
