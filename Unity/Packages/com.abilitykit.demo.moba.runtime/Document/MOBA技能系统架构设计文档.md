## 一、设计理念：统一的技能效果执行模型

### 1.1 传统技能系统的问题

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         传统技能系统的问题                               │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ❌ 技能逻辑耦合严重                                                   │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │  public void ExecuteSkill(Skill skill) {                         │  │
│  │      if (!CheckMana(skill)) return;                            │  │
│  │      if (!CheckCooldown(skill)) return;                         │  │
│  │      PlayAnimation(skill, "precast");                           │  │
│  │      yield return Wait(skill.PrecastDuration);                  │  │
│  │      if (skill.HasProjectile) SpawnProjectile(skill);           │  │
│  │      if (skill.HasDamage) ApplyDamage(skill);                   │  │
│  │      if (skill.HasBuff) AddBuff(skill);                        │  │
│  │  }                                                              │  │
│  │  问题：逻辑堆在一起，改一处动全身                               │  │
│  └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
│  ❌ 不同效果类型独立设计                                              │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │  - 主动技能：有完整的前摇-施法流程                             │  │
│  │  - 被动技能：基于事件触发，逻辑分散                            │  │
│  │  - Buff：独立的生命周期管理                                    │  │
│  │  - 子弹：需要单独的移动和碰撞系统                             │  │
│  │  - AOE：需要区域检测系统                                       │  │
│  │  问题：每个类型都要单独维护，难以复用                         │  │
│  └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
│  ❌ 条件判断分散且难以组合                                            │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │  // 硬编码的条件判断                                           │  │
│  │  if (hp < 30 && hasBuff("Fury") && distance > 10) {            │  │
│  │      ApplyDamage();                                             │  │
│  │  }                                                              │  │
│  │  问题：条件难以复用、难以可视化配置                            │  │
│  └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 二、核心概念：TriggerPlan 统一抽象

### 2.1 TriggerPlan 数据结构

```json
{
  "TriggerId": 10001,
  "EventName": "",
  "EventId": 0,
  "Predicate": {
    "Kind": "And",
    "Nodes": [
      { "Kind": "NumericCompare", "Left": { "Kind": "Resource" }, "Op": "GreaterOrEqual", "Right": { "Kind": "Const", "ConstValue": 100 } },
      { "Kind": "TagCondition", "RequiredTags": ["can_cast"] }
    ]
  },
  "Actions": [
    { "ActionId": "add_buff", "args": { "buff_id": "elite" } }
  ]
}
```

### 3.2 Timeline 与 TriggerPlan 的关系

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    Timeline 与 TriggerPlan 的协作                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  Timeline:  [100ms] ──────→ [200ms] ──────→ [300ms]                   │
│               │               │               │                         │
│               ▼               ▼               ▼                         │
│          EffectId 1      EffectId 2      EffectId 3                     │
│               │               │               │                         │
│               ▼               ▼               ▼                         │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │                   MobaEffectInvoker                               │  │
│  │                        │                                          │  │
│  │                        ▼                                          │  │
│  │  ┌───────────────────────────────────────────────────────────┐  │  │
│  │  │                   TriggerPlan                              │  │  │
│  │  │                                                           │  │  │
│  │  │  Timeline 只负责：在什么时间触发什么 EffectId             │  │  │
│  │  │  TriggerPlan 负责：触发后的条件判断和行为执行             │  │  │
│  │  │                                                           │  │  │
│  │  └───────────────────────────────────────────────────────────┘  │  │
│  └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
│  【重要设计】：Timeline 是纯时间调度，一个 Event = 一个 TriggerPlan      │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │  同一个时间点，判断多种条件？                                    │  │
│  │  ─────────────────────────────────────────                     │  │
│  │  答：不需要多个 Timeline Event，只需要在 TriggerPlan 中用        │  │
│  │      Predicate 的 And/Or/Not 组合即可。                        │  │
│  └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 十一、与 EGameplay 等框架的对比

### 11.1 架构对比表

| 维度 | EGameplay | Ability-Kit |
|------|-----------|------------|
| **设计模式** | Entity-Component (传统 OOP) | ECS + Pipeline + TriggerPlan |
| **配置方式** | Excel/ScriptableObject | JSON + TriggerPlan |
| **执行模型** | Execution + Task | TimelinePhase + TriggerPlan |
| **条件系统** | 内嵌在 Ability 中 | 独立的 Predicate 树 |
| **效果系统** | AbilityEffect | PlanActionModule |
| **条件组合** | 简单表达式 | And/Or/Not 任意嵌套 |
| **复用机制** | 复制粘贴/继承 | TriggerPlan ID 引用 |
| **热更新** | 需要重新编译 | JSON 配置可热更新 |

### 11.2 扩展性对比

**EGameplay 扩展方式：**
```csharp
// 新增效果需要：
1. 创建 Effect 类 (AbilityEffect 子类)
2. 在 Ability 中引用
3. 难以复用
```

**Ability-Kit 扩展方式：**
```csharp
// 新增效果只需：
1. 实现 IPlanActionModule 接口
2. 注册到 PlanActionRegistry
3. JSON 中引用 ActionId，可被多处复用
```

### 11.3 架构优势总结

| 方面 | Ability-Kit 优势 |
|------|---------------|
| **TriggerPlan 统一抽象** | 一次定义，主动/被动/Buff/子弹/AOE 全部复用 |
| **Predicate 配置化** | 复杂条件用 JSON 树表达，无需写代码 |
| **PlanActionModule 热插拔** | 新效果只需注册，不用改核心代码 |
| **条件组合能力** | And/Or/Not 嵌套，无限组合 |
| **复用性** | TriggerPlan 可被多个触发点引用 |
