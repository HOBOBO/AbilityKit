## 一、设计理念：为什么要做技能管线？

### 1.1 传统做法的技能系统问题

```
❌ 传统技能系统的问题：

1. 技能逻辑堆在一个类里
   ┌─────────────────────────────────────────────────────────┐
   │ void ExecuteSkill(Skill skill) {                        │
   │     if (!CheckMana(skill)) return;                     │
   │     if (!CheckCooldown(skill)) return;                 │
   │     if (!CheckRange(skill)) return;                    │
   │                                                          │
   │     // 前摇动画                                          │
   │     PlayAnimation(skill, "precast");                    │
   │     yield return Wait(skill.PrecastDuration);           │
   │                                                          │
   │     // 释放效果                                          │
   │     if (skill.HasProjectile) {                         │
   │         SpawnProjectile(skill);                         │
   │     }                                                   │
   │     if (skill.HasDamage) {                             │
   │         ApplyDamage(skill);                             │
   │     }                                                   │
   │     if (skill.HasEffect) {                             │
   │         ApplyEffect(skill);                             │
   │     }                                                   │
   │                                                          │
   │     // 后摇动画                                          │
   │     PlayAnimation(skill, "postcast");                   │
   │     SetCooldown(skill);                                │
   │ }                                                       │
   │                                                          │
   │ // 新技能要复制粘贴大量代码                             │
   │ // 改一个公共逻辑要改很多地方                          │
   └─────────────────────────────────────────────────────────┘

2. 技能阶段耦合在一起
   ┌─────────────────────────────────────────────────────────┐
   │ // 问题是：检查、动画、效果混在一起                    │
   │ // 想要"改动画不改效果"？很难                         │
   │ // 想要"换目标选择不改释放"？更难                    │
   │                                                          │
   │ // 而且：                                              │
   │ // - 前摇可以被打断                                   │
   │ // - 引导可以被取消                                   │
   │ // - 效果有顺序依赖                                   │
   │ // 这些都很难处理                                    │
   └─────────────────────────────────────────────────────────┘

3. 不同类型技能逻辑差异大
   ┌─────────────────────────────────────────────────────────┐
   │ // 指向技能：直接释放                                  │
   │ // 引导技能：持续施法                                  │
   │ // 弹道技能：发射投射物                                │
   │ // 区域技能：需要目标选择                              │
   │ // 连锁技能：需要找到连锁目标                         │
   │                                                          │
   │ // 每种都要单独处理                                    │
   └─────────────────────────────────────────────────────────┘

4. 测试困难
   ┌─────────────────────────────────────────────────────────┐
   │ // 技能逻辑和游戏状态耦合                              │
   │ // 想要单独测试"释放逻辑"？做不到                    │
   │ // 想要模拟"前摇被打断"？很难                        │
   └─────────────────────────────────────────────────────────┘
```

```

│   │  └─────────────────────────────────────────────┘  │  │
│   │  // 所有逻辑混在一起                               │  │

│   │  │  │ Phase 2: 施法阶段                   │  │  │  │

│   - 可测试                                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 核心理念

```
┌─────────────────────────────────────────────────────────────┐
│                    技能管线的三大承诺                          │
│                                                             │
│   1️⃣ 阶段化                                              │
│      技能释放分为多个阶段，每个阶段职责单一                  │
│                                                             │
│   2️⃣ 可组合                                              │
│      阶段可以复用，技能可以由多个阶段组合而成               │
│                                                             │
│   3️⃣ 可追踪                                              │
│      每个技能有唯一上下文ID，追踪所有触发效果               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、核心概念：从零理解技能管线

### 2.1 什么是"技能管线（SkillPipeline）"？

**通俗解释**：SkillPipeline 是"技能的执行流水线"，把技能释放拆成有序的阶段。

```
┌─────────────────────────────────────────────────────────────┐
│                    技能管线 = 技能的执行流水线               │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐  │
│   │  一个技能 = 一个管线                                  │  │
│   │                                                       │  │
│   │  SkillCastPipeline                                    │  │
│   │  ┌─────────────────────────────────────────────┐   │  │
│   │  │                                               │   │  │
│   │  │   [PreCast]  前摇阶段                       │   │  │
│   │  │   ├─ 条件检查                               │   │  │
│   │  │   ├─ 前摇动画                               │   │  │
│   │  │   └─ (完成后进入 Cast)                       │   │  │
│   │  │                     │                         │   │  │
│   │  │                     ▼                         │   │  │
│   │  │   [Cast]       施法阶段                       │   │  │
│   │  │   ├─ 施法动画                               │   │  │
│   │  │   ├─ 发射投射物                             │   │  │
│   │  │   ├─ 造成伤害                               │   │  │
│   │  │   ├─ 施加buff                               │   │  │
│   │  │   └─ 施法完成                               │   │  │
│   │  │                                               │   │  │
│   │  └───────────────────────────────────────────────┘   │  │
│   │                                                       │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 关键名词解释

| 名词 | 通俗解释 | 类比 |
|------|----------|------|
| **SkillPipelineContext** | 技能执行上下文 | 技能执行时携带的"工作证" |
| **SkillCastContext** | 施法上下文 | 施法时的"状态记录" |
| **SkillTimelinePhase** | 时间轴阶段 | "定时炸弹"，到时间触发 |
| **SkillCastApplyEffectPhase** | 效果应用阶段 | "执行者"，应用效果 |
| **IMobaSkillPipelineLibrary** | 技能库接口 | "技能工厂"，按ID生成技能管线 |
| **SkillExecutor** | 技能执行器 | "指挥官"，发起技能释放 |

## 五、设计模式总结

### 5.1 模块使用的设计模式

```
┌─────────────────────────────────────────────────────────────┐
│                    设计模式应用                              │
│                                                             │
│   1️⃣ Strategy Pattern（策略模式）                          │
│      ─────────────────────────────────────                  │
│      IMobaSkillPipelineLibrary 是技能配置的策略接口         │
│      可以切换不同的实现（硬编码/表驱动）                    │
│                                                             │
│      ┌─────────────────────────────────────────────────┐   │
│      │                                                   │   │
│      │   IMobaSkillPipelineLibrary:                     │   │

│      │   └─► TableDrivenMobaSkillPipelineLibrary      │   │
│      │                                                   │   │
│      └─────────────────────────────────────────────────┘   │
│                                                             │
│   2️⃣ Template Method Pattern（模板方法模式）               │
│      ───────────────────────────────────────────────        │
│      AbilityPipeline 定义管线执行的骨架                      │
│      子类只需实现特定方法                                   │
│                                                             │
│      ┌─────────────────────────────────────────────────┐   │
│      │                                                   │   │
│      │   AbilityPipeline<TCtx>                         │   │
│      │   ├─► Start()  ← 骨架                         │   │
│      │   ├─► Tick()   ← 骨架                         │   │
│      │   └─► 子类实现具体阶段                         │   │
│      │                                                   │   │
│      └─────────────────────────────────────────────────┘   │
│                                                             │
│   3️⃣ Observer Pattern（观察者模式）                        │
│      ─────────────────────────────────────                  │
│      SkillLifecycleEvents 定义生命周期事件                   │
│      观察者可以订阅感兴趣的阶段                            │
│                                                             │
│      ┌─────────────────────────────────────────────────┐   │
│      │                                                   │   │
│      │   SkillLifecycleEvents                          │   │
│      │   ├─► OnPhaseStart                             │   │
│      │   ├─► OnPhaseComplete                          │   │
│      │   └─► OnPipelineComplete                       │   │
│      │                                                   │   │
│      └─────────────────────────────────────────────────┘   │
│                                                             │
│   4️⃣ Context Object Pattern（上下文对象模式）              │
│      ───────────────────────────────────────────────        │
│      SkillPipelineContext 携带所有执行所需的信息            │
│      通过参数传递，而不是全局状态                          │
│                                                             │
│   5️⃣ Cleanup Registration Pattern（清理注册模式）           │
│      ──────────────────────────────────────────────        │
│      RegisterCleanup() 确保资源正确释放                    │
│      即使技能中途失败也能清理                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 各模式解决的问题

| 模式 | 解决的问题 |
|------|------------|
| Strategy | 灵活切换技能配置方式 |
| Template Method | 复用管线执行骨架 |
| Observer | 解耦事件通知 |
| Context Object | 避免全局状态 |
| Cleanup Registration | 确保资源释放 |

### 6.1 基础技能实现示例

```csharp
// 场景：实现一个简单的伤害技能

public class SimpleDamageSkillExecutor
{
    private readonly IMobaSkillPipelineLibrary _library;
    private readonly SkillPipelineRunner _runner;

    public void ExecuteSkill(int skillId, int casterId, int targetId)
    {
        // 创建施法请求
        var request = new SkillCastRequest
        {
            SkillId = skillId,
            CasterActorId = casterId,
            TargetActorId = targetId,
            AimPos = Vec3.Zero,
            AimDir = Vec3.Forward
        };

        // 查询技能配置
        if (!_library.TryGet(skillId,
            out var preConfig, out var prePhases,
            out var castConfig, out var castPhases))
        {
            return;  // 技能不存在
        }

        // 创建上下文
        var ctx = new SkillPipelineContext(request);

        // 启动管线
        _runner.Start(preConfig, prePhases, castConfig, castPhases,
            abilityInstance: null, ctx, triggerContext, out _);
    }
}
```

### 6.3 自定义阶段示例

```csharp
// 场景：实现一个"锁定目标"阶段

public sealed class LockTargetPhase :
    AbilityInstantPhaseBase<SkillPipelineContext>
{
    public LockTargetPhase(
        AbilityPipelinePhaseId phaseId,
        string targetKey)
    {
        PhaseId = phaseId;
        TargetKey = targetKey;
    }

    public string TargetKey { get; }

    protected override void OnInstantExecute(SkillPipelineContext context)
    {
        // 从共享数据获取目标
        if (!context.SharedData.TryGetValue(TargetKey, out var targetId))
        {
            // 目标不存在，失败
            context.IsAborted = true;
            return;
        }

        // 锁定目标
        var target = ResolveActor((int)targetId);
        context.SharedData["LockedTarget"] = target;

        // 注册清理
        context.RegisterCleanup(() => target.Unlock());
    }
}
```

### 6.4 条件检查阶段示例

```csharp
// 场景：实现能量检查阶段

public sealed class EnergyCheckPhase :
    AbilityInstantPhaseBase<SkillPipelineContext>
{
    protected override void OnInstantExecute(SkillPipelineContext context)
    {
        var caster = ResolveActor(context.CasterActorId);
        var energy = caster.Attributes.Get(AttributeType.Energy);
        var cost = GetSkillEnergyCost(context.SkillId);

        if (energy.CurrentValue < cost)
        {
            context.IsAborted = true;
            context.FailReason = "Not enough energy";
            return;
        }

        // 消耗能量
        energy.RemoveValue(cost);
    }
}
```
